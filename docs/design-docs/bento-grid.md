# Бенто-сетка: дизайн движка

## Проблема

Адаптивная сетка, где блоки имеют разные размеры (1×1, 2×2 и т.д.) по трём брейкпоинтам. Пользователь может перетаскивать для изменения порядка и ресайзить. Layout должен персистироваться между сессиями и синхронизироваться с публичным видом.

## Решение

Два JSON-поля на `Profile`:

- `blockSizes` — размеры каждого блока: `Record<blockId, BlockGridSize>`
- `layout` — порядок колонок по брейкпоинтам: `{ mobile, tablet, desktop: number[][] }`

Движок (`lib/block-grid.ts`) вычисляет CSS grid placement из этих полей в момент рендера.

---

## Система якорей

Каждый блок может иметь явные позиции-якоря по брейкпоинтам (`anchorsByBreakpoint`).

Если якорь отсутствует — **sparse-anchor assignment**: блок размещается в первый свободный слот после предыдущего блока.

**Разрешение перекрытий:** если два блока претендуют на одну ячейку, побеждает тот, у которого меньший индекс в `layout`; другой сдвигается вниз.

---

## Clamping

Блоки зажимаются по границам сетки:
- `colSpan` не может превышать `GRID_COLUMNS - gridColumnStart + 1`
- Это предотвращает выход блоков за правую границу контейнера

---

## Поведение по брейкпоинтам

- `mobile` / `tablet`: 2 колонки. Используется один и тот же массив `layout`.
- `desktop`: 4 колонки.

При ресайзе редактор сохраняет оба поля через `PATCH /api/profile`.

---

## Section-блоки

`section`-блоки имеют фиксированную высоту 56px вне зависимости от `rowSpan`. Всегда занимают полную ширину сетки (`colSpan = GRID_COLUMNS`).

---

## Почему не CSS Masonry

Нативный CSS Masonry (или библиотеки типа Masonry.js) не дают контроля над позицией конкретного блока — нужна возможность drag-to-reorder с сохранением позиции. Поэтому используется CSS Grid с явными `grid-column` / `grid-row` на каждый блок.

---

## Единицы

- `GRID_COLUMNS` — 2 (mobile/tablet), 4 (desktop)
- `BENTO_ROW_UNIT` — 8px (одна единица высоты ряда)
- `rowSpan` — от 1 до 6; итоговая высота = `rowSpan * BENTO_ROW_UNIT * <row-height-factor>`
