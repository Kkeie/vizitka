# Объяснение автотестов простым языком

Этот текст написан **для любого человека**, не только для программистов. Здесь объясняется, *зачем* мы гоняем тесты и *что именно* они проверяют.

---

## Что вообще происходит

Представьте **автоматического тестировщика**: он не открывает сайт глазами, а шлёт запросы на сервер так же, как мобильное приложение или страница в браузере. Если ответ **не такой, как ожидали** (например, вместо «успех» пришла ошибка), тест **падает** — это сигнал, что что-то сломали в коде.

- **Тесты API** (папка `backend/test/`) проверяют **сервер**: регистрация, профиль, карточки, загрузки и т.д.
- **Тесты интерфейса** (папка `frontend/e2e/`) запускают **настоящий браузер**: нажимают кнопки «Войти», «Создать аккаунт» и смотрят, куда перешла страница.

Запуск (как в проекте уже принято):

- Серверные тесты: в каталоге `backend` команда `npm test`.
- Браузерные тесты: в каталоге `frontend` команда `npm run test:e2e` (нужен установленный Chromium для Playwright).

---

## Как читать названия тестов

В названии часто есть код вроде **TC-AUTH-01** — это ссылка на строку в файле `docs/TEST-CASES.md` (официальный список сценариев).

Дальше в том же названии обычно **простое описание по-русски**: что должно получиться «с точки зрения человека» (например: «не пускает без пароля»).

## Словарь (если встретили в логах)

| Термин | Простыми словами |
|--------|------------------|
| **Токен** | Длинная строка-«пропуск»: с ней сервер понимает, кто вы. В браузере он хранится в `sessionStorage`. |
| **401 / unauthorized** | «Пропуска нет или он поддельный» — как закрытая дверь без ключа. |
| **404 / not_found** | «Такой страницы или пользователя нет». |
| **400** | «Запрос составлен неправильно» — например, не хватает поля или ник слишком короткий. |
| **409** | «Такой ник уже занят» при регистрации. |
| **413** | «Файл слишком тяжёлый» для загрузки. |
| **JSON** | Удобный для программ формат ответа; тесты проверяют, что ошибка приходит **текстом в JSON**, а не белой страницей HTML. |

## Соответствие документу TEST-CASES.md

Основные коды **TC-*** закрыты тестами из таблиц в `TEST-CASES.md`. Часть сценариев продублирована **дополнительными** тестами в файлах `*.more.integration.test.ts`: там проверяются похожие ситуации «в быту» (регистр букв, пустые поля, три карточки вместо двух и т.д.).

**Интерфейс:** автоматизированы типовые действия (регистрация, вход, выход, заметка в редакторе, публичная страница). Сценарии вроде «перетащить карточку мышкой» или «выбрать файл в окне Windows» сложнее автоматизировать без специальных приёмов — их по-прежнему можно прогонять вручную по `TEST-CASES.md`.

---

## Зачем это всё обычному человеку

Когда тесты зелёные, это значит: **основные сценарии, по которым люди реально пользуются сервисом, сейчас работают так, как описано**. Если что-то сломают при правке кода, тест чаще всего **сразу покажет**, что именно перестало совпадать с ожиданиями — не нужно вручную кликать десять разных экранов перед каждым релизом.
